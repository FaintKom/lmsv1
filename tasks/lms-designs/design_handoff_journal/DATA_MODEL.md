# GrassLMS · Модуль «Журнал» — модель данных и обработка

_Документ-обоснование к прототипам админ-модуля. Описывает сущности, связи,
как ловить пересечения кабинетов и как считать движение группы по программе
(pacing). Учитывает рост со средней школы до больших (сотни групп / филиалы)._

---

## 0. Главный принцип: разделяем «план» и «факт»

Почти вся боль текущего прототипа — оттого, что в одной таблице смешаны три
разные вещи. Их нужно развести на три слоя:

| Слой | Что это | Меняется |
|---|---|---|
| **Шаблон / план** | недельная сетка занятий, программа курса (scope & sequence) | редко, заранее |
| **Экземпляры (факт-календарь)** | конкретные занятия на даты, сгенерированные из шаблона | при изменении расписания |
| **Факт проведения** | посещаемость, что реально прошли, что сделали студенты | каждый урок |

Расписание = «план». Журнал-матрица = «факт проведения». Pacing = «план VS факт».
Каждый экран читает свой слой, поэтому ничего не «свалено в кучу».

## 1. Сущности (ER)

```
ORG ──< SITE (филиал) ──< ROOM
ORG ──< USER (роль: admin | methodist | teacher | student)
ORG ──< COURSE ──< GROUP ──< ENROLLMENT >── STUDENT(USER)
                    │
GROUP ──< SCHEDULE_SLOT (шаблон: день недели · время · room)
GROUP ──< SESSION (экземпляр: дата · время · room · held)
COURSE ──< CURRICULUM_TOPIC (план программы, упорядочен)
SESSION ──> CURRICULUM_TOPIC (actual_topic_id — что реально прошли)
SESSION ──< ATTENDANCE (по студенту: статус P/L/A/E)
SESSION ──< ACTIVITY (что студент сделал: попытки/завершения упражнений)
ROOM может быть kind=offline (физический) | online (виртуальный, как в Google Calendar)
```

### 1.1 Справочники

**`sites`** — филиалы. `id, org_id, name, timezone`.
Нужны сразу, даже если филиал один: большие школы = несколько площадок, и
пересечения кабинетов считаются **в рамках одной площадки**.

**`rooms`** — кабинеты И онлайн-комнаты в одной таблице.
```
id, org_id, site_id (null для online),
name,            -- "Каб. 7", "Zoom-1", "Jitsi-A"
kind,            -- 'offline' | 'online'
capacity,        -- вместимость (offline) / лимит (online, может быть null)
meeting_url,     -- только online: постоянная ссылка на встречу
color,           -- для сетки
is_active
```
Решение из ответов: **из одного кабинета не бывает одновременно онлайн и офлайн**
— поэтому формат занятия определяется `room.kind`, а не флагом на занятии.
Онлайн-комнаты тоже бронируются и тоже могут пересекаться (один Zoom-аккаунт =
один поток), поэтому детект пересечений работает одинаково для обоих kind.

**`users`** — все люди. `id, org_id, role, full_name, email`. Преподаватель
привязан к группам через `group.teacher_id` (или many-to-many, если ведут вдвоём).

### 1.2 Учебные сущности

**`courses`** — предмет/программа. `id, org_id, title, subject, level`.

**`groups`** — конкретная группа студентов по курсу.
```
id, course_id, site_id, name,   -- "Math 3A", "Coding 1 / вечер"
teacher_id, default_room_id,
start_date, end_date, status    -- planned | active | archived
```

**`enrollments`** — кто в группе. `id, group_id, student_id, status (active/left), joined_at`.
M:N студент↔группа. Один студент может быть в нескольких группах.

### 1.3 Расписание: шаблон и экземпляры

**`schedule_slots`** — недельный **шаблон** группы (повторяющееся правило).
```
id, group_id, room_id,
weekday,         -- 1..7
start_time, end_time,   -- локальное время площадки
valid_from, valid_until -- период действия правила (для смены расписания с даты)
```
Один слот = «эта группа по понедельникам 09:00–10:00 в каб. 7».

**`sessions`** — **экземпляры** занятий на конкретные даты. Генерируются из слотов
на диапазон дат («сгенерировать журнал»), но дальше живут самостоятельно
(можно перенести/отменить одно занятие, не трогая шаблон).
```
id, group_id, room_id, slot_id (origin, nullable),
date, start_time, end_time,
held,                 -- bool: проведено ли
actual_topic_id,      -- FK → curriculum_topics (что реально прошли)
planned_topic_id,     -- снимок плана на момент генерации (для дрейфа)
note, canceled, created_by
```
**Почему отдельные `sessions`, а не вычислять «на лету» из слотов:** к занятию
цепляется факт (посещаемость, тема, активность). Факт можно повесить только на
устойчивый id, а не на правило-повторение. Это же даёт историю при смене
расписания.

### 1.4 Программа курса (scope & sequence)

**`curriculum_topics`** — упорядоченный план тем курса.
```
id, course_id,
position,        -- порядок 1..N
title,           -- "Дроби — введение"
planned_lessons, -- сколько занятий заложено на тему (для pacing по «весу»)
target_date      -- (опц.) плановая дата прохождения, для pacing по календарю
```
Это шаблон уровня **курса**, общий для всех его групп. Каждая группа движется по
нему в своём темпе — её позиция вычисляется из `sessions.actual_topic_id`.

### 1.5 Факт урока

**`attendance`** — посещаемость, одна строка на (session × student).
```
id, session_id, student_id,
status,   -- 'present' | 'late' | 'absent' | 'excused'  (P/L/A/E)
marked_by, marked_at
```
Уникальный индекс `(session_id, student_id)` — UPSERT при отметке, без дублей.

**`activity`** — «что студент реально сделал на уроке». Не выдумываем числа: это
агрегат из существующих попыток упражнений LMS, привязанный к занятию по
(student_id, date).
```
id, session_id, student_id,
exercises_done, exercises_total,
items_correct, time_spent_sec,
xp_earned, last_event_at
```
Источник — таблица попыток упражнений (`exercise_attempts`), которая уже есть в
LMS. `activity` — это материализованная свёртка для быстрого показа в журнале.

## 2. Детект пересечений кабинетов

Пересечение = **две session-и в одном `room_id`, в одну `date`, с
перекрывающимися интервалами времени**. (Для шаблона — то же самое на уровне
слотов в рамках одной недели.)

### 2.1 Правило перекрытия интервалов
Два интервала `[a_start, a_end)` и `[b_start, b_end)` пересекаются, если:
```
a_start < b_end  AND  b_start < a_end
```
(полуинтервалы: занятие 09:00–10:00 и 10:00–11:00 НЕ конфликтуют).

### 2.2 Где считать — три уровня, разная стоимость

1. **На сохранении слота/занятия (обязательно).** Перед записью —
   запрос «есть ли в этом room+date другой session с перекрытием». Если есть →
   `409 Conflict`, в ответе список конфликтующих. UI даёт «Сохранить всё равно»
   (`force=true`) — иногда это нужно (две подгруппы в большом зале).
   ```sql
   SELECT s.* FROM sessions s
   WHERE s.room_id = :room AND s.date = :date AND s.id <> :self
     AND s.start_time < :end AND :start < s.end_time
     AND s.canceled = false;
   ```
   Индекс: `(room_id, date, start_time)`.

2. **Для сетки кабинетов (обзор на день/неделю).** Тянем все session-и площадки
   за период одним запросом, группируем по room → сортируем по времени →
   линейным проходом отмечаем перекрытия (sweep line). O(n log n), без N+1.

3. **Фоном (для больших).** При сотнях групп держим
   `room_occupancy(room_id, date, start, end, session_id)` как
   денормализованную проекцию и пересчитываем только затронутый
   (room, date) при изменении. Сетка тогда читается мгновенно.

### 2.3 Что считать конфликтом помимо времени
- **Вместимость**: `enrollments_active > room.capacity` → мягкое предупреждение
  (не блок).
- **Двойная занятость преподавателя**: тот же `teacher_id` в перекрытии — тоже
  конфликт, ловится тем же sweep-проходом по другому ключу.
- **Площадка**: онлайн-комнаты site_id=null → пул общий по орг; офлайн —
  в пределах site.

## 3. Движение группы по программе (pacing)

Вопрос «отстаём / идём ровно / опережаем?» для каждой группы.

### 3.1 Что знаем
- План: `curriculum_topics` курса (упорядочены, у каждой `planned_lessons` и опц.
  `target_date`).
- Факт: `sessions` группы с `held=true` и `actual_topic_id`.

### 3.2 Позиция группы
```
covered_topics   = distinct actual_topic_id у проведённых session-ий группы
current_topic    = тема последнего проведённого занятия
next_topic       = первая тема по position, которой нет в covered
progress %       = covered_topics.count / curriculum_topics.count
```

### 3.3 Темп: на сколько отстаём/опережаем — два режима

**Режим A — по количеству занятий (есть всегда).**
```
expected_pos = сумма planned_lessons по темам, которые должны быть пройдены
               к этому моменту (по числу проведённых занятий группы)
delta = covered_lessons_actual − expected_lessons
```

**Режим B — по календарю (если заданы `target_date`).**
```
on_schedule_topic = последняя тема, чей target_date <= сегодня
delta = position(current_topic) − position(on_schedule_topic)
```

**Бэйдж:**
```
delta >=  +1 тема → "Опережает"   (sun)
|delta| <  1      → "Идёт ровно"  (green)
delta <=  -1 тема → "Отстаёт"     (coral),  показываем «на N тем / N занятий»
```

### 3.4 Дрейф плана vs факта
На занятии `planned_topic_id` (снимок на момент генерации) сравнивается с
`actual_topic_id`. Расхождение = группа отклонилась от плана на этом уроке —
подсветка в детали занятия и на таймлайне.

### 3.5 Производительность
Pacing-борд по всем группам = для каждой группы 2 числа (covered, expected).
Считается одним сгруппированным запросом по `sessions` + кэш счётчика
`group.covered_topics_count` (инкремент при простановке `actual_topic_id`).
Для больших школ — ночной пересчёт + инкремент на запись.

## 4. Быстрая простановка посещаемости (боль «много кликов»)

Модель это уже позволяет; дело в UI + API:
- **UPSERT-батч**: `PUT /sessions/:id/attendance` принимает массив
  `[{student_id, status}]` — один запрос на весь класс, а не по клику.
- **«Все присутствуют» = дефолт.** Создаём строки present для всех enrollment,
  преподаватель только снимает отсутствующих. На клавиатуре: ↑/↓ по списку,
  P/L/A/E или 1/2/3/4 — статус, Enter — следующий.
- **Оптимистичный UI**: статус применяется мгновенно, синк в фоне.
- Матрица-история читается из `attendance` за период (students × dates),
  цвет ячейки = status. Редактирование прямо в ячейке = тот же UPSERT.

## 5. Роли и доступ (один модуль, три роли)
| Может | admin/координатор | методист | преподаватель |
|---|---|---|---|
| Расписание, кабинеты, слоты | ✅ | 👁 чтение | 👁 свои группы |
| Журнал-матрица | вся орг | вся орг | свои группы |
| Простановка посещаемости | ✅ | ✅ | ✅ свои |
| Программа / pacing | ✅ | ✅ (главный потребитель) | свои группы |
| Справочники (rooms, courses) | ✅ | — | — |

Один набор экранов, фильтры и объём данных меняются от роли (scope), а не верстка.

## 6. Что ещё понадобится (предлагаю, не делаю без подтверждения)
- **Замены и переносы**: отмена занятия + перенос на дату/в кабинет с
  пере-проверкой конфликтов; журнал замен преподавателя.
- **Праздники / нерабочие дни** на уровне орг и site — генератор занятий их
  пропускает.
- **Уведомления**: студенту/родителю при отсутствии; преподавателю — «журнал не
  заполнен за вчера».
- **Экспорт**: посещаемость в CSV/XLSX за период (для отчётности).
- **Аудит**: кто и когда менял посещаемость/расписание.
